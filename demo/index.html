When to use

wc-speech fits sites that want an optional in-page listen control for sighted visitors who do not already use text-to-speech software. It reads content you point target at, follows lang attributes, and can highlight the current sentence and word when the browser supports it.

Good fit

  • Public information pages where a spoken version helps visitors who would not otherwise have one.
  • Articles, guides, or policy text where an explicit "Listen" or speech toolbar is welcome.
  • Projects that can self-host a few static files (wc-speech.js, speech.css) with no build step or npm dependency.
  • Pragmatic coverage of "written and oral format" expectations — see Why it exists.

Not a good fit

  • Primary accessibility compliance. Use semantic HTML, keyboard support, and assistive-technology testing instead. This component is intentionally hidden from screen readers (aria-hidden="true" on the host).
  • Replacing screen readers or OS narration. People who rely on those tools already have more capable speech; they will not use a per-site toolbar.
  • Guaranteed legal compliance. It may help with oral access to written content but is not a substitute for a full accessibility programme or professional audit.
  • Offline or server-side TTS. Everything runs in the visitor's browser via speechSynthesis; quality and voices depend on their device.

Compared with other options

Approach When it wins When wc-speech wins
Screen reader / OS read-aloud User needs site-wide, reliable narration; accessibility is the goal. You want an optional on-page button for sighted users without TTS installed.
Browser "Read aloud" / Immersive Reader User already uses a built-in tool; no site integration needed. You want speech tied to your page layout, highlights, and sentence navigation.
Cloud TTS API (Google, Amazon, etc.) Consistent voice quality, offline playback, or languages the browser lacks. Zero cost, no API keys, no audio files — browser voices only.
Roll your own speechSynthesis wrapper Minimal needs, full control, no DOM walking. You want sentence splitting, lang runs, abbr expansion, highlights, and browser workarounds out of the box.

Production checklist

  • Set lang on <html> and on mixed-language sections.
  • Provide a fallback when data-speech-state="unsupported" — hide or disable the speech control.
  • Keep readable content in target; put forms and the speech toolbar outside it when you can.
  • Override error-* and label-* attributes if your site is not in English.
  • Test with system voices on your target platforms (see Troubleshooting).

Documentation

wc-speech is a small text-to-speech web component built on the Web Speech API. It reads text from another part of the page, follows the document's lang attributes, and highlights the current sentence and word when the browser supports custom highlights.

Known limits

See also the README in the repository.

  • One active instance per page (enforced) — only the first connected <wc-speech> runs. Additional instances are disabled (data-speech-blocked="duplicate") until the active one is removed.
  • Point target at readable content. wc-speech subtrees are skipped automatically; keep other controls outside the target when you can.
  • When speech synthesis is unsupported, controls are disabled and a message is shown in [data-speech-error] when present.
  • Pause cancels the current sentence; resume starts that sentence again from the beginning.
  • Word highlighting depends on browser, voice, and the Custom Highlight API.
  • Automated tests run in CI — run npm test locally before changing speech behaviour.

Who this is for

wc-speech is a simple in-page speech tool for sighted users who do not have text-to-speech software installed. It is not a replacement for assistive technology. People who rely on screen readers or built-in operating-system narration already have their own, more capable speech tools and will use those — this component is intentionally simple and stays out of their way.

Why it exists (Directive (EU) 2019/882)

The European Accessibility Act (Directive (EU) 2019/882) Article 13.2 encourages key information to be available through more than one sensory channel — for instance, text that can also be rendered orally. wc-speech provides an in-page spoken version of written content to help cover that "written and oral format" expectation for visitors who would not otherwise have a speech option.

That said, native browser and operating-system text-to-speech features — and dedicated assistive technology — generally do this more reliably than a per-site speech tool. Treat wc-speech as a pragmatic fill for a specific gap, not as a substitute for real assistive technology or a guarantee of legal compliance.

Expected markup

Inside <wc-speech>, wire voice, speed, and scroll with data-speech-* hooks on a select, form control, and checkbox. Add a [popover] options panel inside a speech bar (.speech-bar or [data-speech-bar]), command buttons with commandfor and command, and optionally [data-speech-error]. Play/pause buttons may sit inside or outside <wc-speech>; adjustment controls should stay inside. See the advanced demo toolbar, or the simple demo for a one-button setup.

Basic setup

Add the component, point target at the content that should be read, and wire buttons to the component with commandfor and command. 

<button type="button" commandfor="speech" command="--show-controls">
  Show speech tool
</button>

<wc-speech
  id="speech" target="#article" prefer-voice="Microsoft" hidden aria-hidden="true">

  <section class="speech-bar" hidden>
    <button type="button" commandfor="speech" command="--playpause">
      <span data-speech-face="play">Play</span>
      <span data-speech-face="pause" hidden>Pause</span>
    </button>

    <button type="button" commandfor="speech"
      command="--previous-sentence">Previous</button>

    <button type="button" commandfor="speech"
      command="--next-sentence">Next</button>

    <select data-speech-voice></select>

    <input data-speech-rate type="range" min="0" max="3.5" step="0.5" value="1">
  </section>
</wc-speech>

<article id="article" lang="en">
  <h1>Readable content</h1>
  <p>This text can be spoken.</p>
</article>

Attributes

target
CSS selector for the content that should be read. Put controls outside the target when you can. wc-speech elements inside the target are skipped automatically.
data-speech-voice
On a <select> inside <wc-speech>. Populated with available voices. Use prefer-voice on the component to auto-select a matching voice.
data-speech-rate
On a form control inside <wc-speech> whose value is a number — for example input type="range", input type="number", or select with numeric option values. Read when each sentence is spoken; values are clamped between 0.1 and 10.
data-speech-scroll
On <input type="checkbox"> inside <wc-speech>. Toggles the component scroll attribute when checked.
prefer-voice
Optional text used to prefer matching voice names or voice URIs when the component auto-selects a voice, for example Microsoft for system voices on Windows. Users can still choose another voice manually.
scroll
Optional boolean. When present, the page follows along by scrolling the current sentence into view while reading. Honors prefers-reduced-motion (smooth scrolling is disabled when the user prefers reduced motion). This attribute is the default for the "Scroll to follow while reading" checkbox in the voice options, and users can toggle it there at runtime.
code-lang
Optional language used when reading pre and code blocks, since source code is normally English. Defaults to en and ignores the surrounding document language. An explicit lang attribute on the code element still wins.
label-play / label-pause
Optional accessible name for the play/pause button (used as aria-label and title unless the button has data-speech-manual-a11y). Defaults to Play and Pause. Visible icon and text live in your markup — see data-speech-face below.
data-speech-face="play" / data-speech-face="pause"
On child elements of a command="--playpause" or command="--speech-marked" button. The component toggles which face is shown and sets data-speech-action on the button. You supply icons, emoji, or text; wc-speech does not hardcode them. Icon-only buttons can use aria-hidden="true" on faces and rely on label-play / label-pause for the accessible name.
status-speaking / status-paused / status-finished
Optional text for a [role="status"] hook when present. Defaults to Speaking, Paused, and Finished. Has no effect on assistive technology when aria-hidden="true" is on <wc-speech> (the default after connect).
error-unsupported / error-missing-lang / error-missing-target / error-target-not-found / error-empty-content / error-synthesis-failed
Optional user-visible error text for common failure cases. Override these attributes for translation. Errors are logged to the console, dispatched as speech-error, and shown in [data-speech-error] when present.
data-speech-state
Set on the host element: ready, speaking, paused, unsupported, or error. Use for styling, for example wc-speech[data-speech-state="error"].
hidden
On <wc-speech>, a pre-init cloak removed when the component connects. On the speech bar (.speech-bar or [data-speech-bar]), hides the toolbar until --show-controls.
aria-hidden
Set to true when the component connects so screen readers and other assistive technology stay out of the speech tool. Sighted users interact with visible controls; the tool is not a replacement for assistive technology.
[popover][role="toolbar"]
Optional selection toolbar inside <wc-speech>. Shown above text the user selects in the target content. Wire a button to command="--speech-marked" to read only the marked text using the same DOM collection, language, and abbreviation rules as full-page speech. Use popover="auto" for light dismiss on click outside or Escape. Distinct from the voice options [popover] panel.

Commands

--show-controls / --hide-controls
Show or hide the control bar. Hiding also stops speech and clears highlights.
--playpause
Start speech, pause the current sentence, or continue from the start of the current sentence. Pause uses cancel() rather than pause() for better cross-browser behaviour.
--previous-sentence / --next-sentence
Move backward or forward while speech is active.
--toggle-options
Show or hide the voice and speed options popover.
--speech-marked
Read only the text currently selected in the target content. Used from the selection toolbar ([popover][role="toolbar"]). While marked speech is active, repeated clicks pause or resume the current sentence (same behaviour as --playpause). If full-page speech is active, switches to the marked selection instead.

Keyboard

Escape
Closes the selection toolbar or voice options popover when open. While speech is active, stops reading and clears highlights (same as hiding the control bar) once overlays are closed.

Events

The component dispatches bubbling, composed CustomEvents you can listen for: speech-start, speech-stop, speech-finish, speech-error, and sentence-change. The speech-error event includes detail.code, detail.message, and detail.index.

Language

Add a lang attribute to the page and to any content that changes language. The component resolves language from the element being spoken and its ancestors, so mixed-language content like <span lang="fr"> can use another voice when available.

Highlight colors

Override these CSS custom properties to change the word and sentence highlights: 

:root {
  --speech-word-bg: Mark;
  --speech-word-text: MarkText;
  --speech-sentence-bg: SelectedItem;
  --speech-sentence-text: SelectedItemText;
}

Without the Custom Highlight API, img, video, and audio use an element outline while they are spoken. abbr, time, code, and pre use the sentence background on the element instead, like other inline text.

Worth knowing

  • Only one <wc-speech> per page is active. Additional instances are disabled with data-speech-blocked="duplicate". When the active instance is removed, the next blocked instance is promoted.
  • The component needs browser support for speechSynthesis and SpeechSynthesisUtterance.
  • Word and sentence highlighting is progressive enhancement. Speech still works without the Custom Highlight API.
  • Word highlighting also depends on the selected voice. Some Chrome Google voices do not emit reliable word boundaries, so the component may fall back to sentence highlighting only. Use prefer-voice to prefer voices that behave better for your target browsers.
  • Native commandfor support is still new, so the component includes a click fallback for its own commands.
  • Image alt text is spoken as-is; empty alt is skipped. abbr title is read as the expanded form. time datetime can add a spoken date. pre and code are read in code-lang. wc-speech, form controls, script, style, and elements with aria-hidden="true" or hidden are skipped.

Troubleshooting

Common symptoms mapped to causes and fixes. Check data-speech-state on <wc-speech>, the browser console, and [data-speech-error] when present. Listen for speech-error events — detail.code matches the error attribute names below.

No voices in the dropdown

Symptom: [data-speech-voice] is empty or only shows a few voices.
Cause: The browser exposes only voices installed on the visitor's device.
Fix: Install system voices — see Windows or Mac. Use prefer-voice="Microsoft" (or similar) to auto-select a system voice when available.

Speech control disabled / "not supported"

Symptom: Buttons stay disabled; data-speech-state="unsupported".
Cause: error-unsupported — no speechSynthesis in this browser.
Fix: Hide the speech tool or show a static message. Speech cannot run without browser support.

Nothing is spoken

Symptom: Play starts but no audio, or speech stops immediately.
Possible causes:

  • error-missing-lang — add lang to <html>.
  • error-missing-target or error-target-not-found — check the target selector matches an element on the page.
  • error-empty-content — readable text is missing inside target (controls and aria-hidden subtrees are skipped).
  • Content loaded after connect — the component collects sentences when speech starts; ensure the DOM is ready before the user presses play.

Wrong language or voice

Symptom: English text read with the wrong accent, or mixed-language content sounds off.
Cause: Missing or incorrect lang attributes; no matching voice for that language.
Fix: Set lang on <html> and on inline changes (for example <span lang="da">). Install voices for those languages. The component picks a voice per language run when one is available.

Word highlights missing

Symptom: Sentences highlight but individual words do not.
Cause: Word boundaries depend on browser, voice, and the Custom Highlight API. Some Chrome Google voices never emit reliable word events.
Fix: Try a system voice via prefer-voice. Sentence-level highlighting still works; speech is unaffected.

Pause restarts the current sentence

Symptom: After pause, the same sentence plays from the beginning.
Cause: Expected behaviour — pause uses cancel() because speechSynthesis.pause() is unreliable in several browsers.
Fix: No fix needed; document this for your users if you provide a help page.

Speech cuts off mid-sentence (Chrome)

Symptom: Long sentences stop partway through.
Cause: Chrome can silently stop a single utterance after ~15 seconds.
Fix: The component uses a keep-alive heartbeat and splits content into sentences. If you still hear cut-offs, check for very long unbroken text without sentence punctuation.

Second wc-speech does nothing

Symptom: Another instance on the page is inert; console warns about duplicates.
Cause: Only one active instance per page (data-speech-blocked="duplicate").
Fix: Use a single <wc-speech> per page, or remove the active one so the next is promoted.

error-synthesis-failed during playback

Symptom: Reading stops; error message or speech-error event.
Cause: The selected voice or browser failed to synthesize an utterance.
Fix: Ask the user to pick another voice in [data-speech-voice]. Retry in another browser if the problem persists.
